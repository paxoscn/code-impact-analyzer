# Java 字段修改检测功能说明

## 功能概述

在分析 Git patch 文件时，系统现在能够自动识别 Java 类属性（字段）的修改，并将其视为对相应 getter 和 setter 方法的修改。这使得影响分析更加全面和准确。

## 核心特性

### 1. 自动方法生成

当检测到字段修改时，系统会自动生成对应的访问器方法：

| 字段类型 | 字段名 | Getter 方法 | Setter 方法 |
|---------|--------|------------|------------|
| String | userName | getUserName() | setUserName(String) |
| int | age | getAge() | setAge(int) |
| boolean | isActive | isActive() | setActive(boolean) |
| List\<String\> | items | getItems() | setItems(List\<String\>) |

### 2. Boolean 字段特殊处理

对于以 `is` 开头的 boolean 字段：
- **Getter**: 保持原样，如 `isActive()`, `isValid()`
- **Setter**: 移除 `is` 前缀，如 `setActive(boolean)`, `setValid(boolean)`

这符合 Java Bean 规范的标准命名约定。

## 使用场景

### 场景 1: 字段重命名

```java
// 修改前
private String userName;

// 修改后
private String username;
```

**影响分析结果:**
- `getUserName()` - 可能需要更新
- `setUserName(String)` - 可能需要更新
- `getUsername()` - 新方法
- `setUsername(String)` - 新方法

### 场景 2: 字段类型变更

```java
// 修改前
private List<String> items;

// 修改后
private List<Item> items;
```

**影响分析结果:**
- `getItems()` - 返回类型可能变化
- `setItems(List<String>)` - 旧方法
- `setItems(List<Item>)` - 新方法

### 场景 3: 新增字段

```java
// 新增
private String description;
```

**影响分析结果:**
- `getDescription()` - 新方法
- `setDescription(String)` - 新方法

## 编程接口

### Rust API

```rust
use code_impact_analyzer::patch_parser::PatchParser;
use std::path::Path;

fn analyze_field_changes(patch_path: &Path) -> Result<(), Box<dyn std::error::Error>> {
    // 解析 patch 文件
    let changes = PatchParser::parse_patch_file(patch_path)?;
    
    // 遍历每个文件的变更
    for change in &changes {
        // 提取字段修改对应的方法
        let methods = PatchParser::extract_modified_field_methods(change);
        
        if !methods.is_empty() {
            println!("文件: {}", change.file_path);
            println!("受影响的方法:");
            for method in methods {
                println!("  - {}", method);
            }
        }
    }
    
    Ok(())
}
```

## 技术实现

### 检测算法

1. **解析 Patch 文件**
   - 使用 `gitpatch` crate 解析 Git diff 格式
   - 提取每个文件的变更块（hunks）

2. **识别字段声明**
   - 分析添加/删除的代码行
   - 使用模式匹配识别字段声明
   - 提取字段类型和名称

3. **生成方法名**
   - 根据 Java Bean 规范生成 getter/setter
   - 特殊处理 boolean 类型字段
   - 保留泛型类型信息

4. **去重和排序**
   - 合并重复的方法名
   - 按字母顺序排序输出

### 字段识别规则

系统能够识别以下格式的字段声明：

```java
// 基本字段
private String name;
public int age;
protected boolean isActive;

// 带初始值
private String name = "default";
private int count = 0;

// 静态常量
public static final int MAX_SIZE = 100;
private static final String PREFIX = "USER_";

// 泛型类型
private List<String> items;
private Map<String, Object> metadata;

// 数组类型
private String[] tags;
private int[][] matrix;
```

### 排除规则

以下内容不会被识别为字段：

```java
// 方法声明
public void setName(String name) { }
private String getName() { }

// 类声明
public class User { }
interface UserService { }

// 注解
@Override
@Autowired

// 导入语句
import java.util.List;

// 注释
// private String comment;
/* private String comment; */
```

## 常见问题

### Q1: 为什么需要这个功能？

**A:** 在 Java 开发中，字段通常通过 getter/setter 方法访问。当字段被修改时，相关的访问器方法也会受到影响。这个功能帮助你自动识别这些潜在的影响点。

### Q2: 是否支持 Lombok 注解？

**A:** 目前不支持。如果类使用了 `@Data`, `@Getter`, `@Setter` 等 Lombok 注解，系统无法自动识别生成的方法。这是未来的改进方向。

### Q3: 如何处理自定义的 getter/setter 命名？

**A:** 目前只支持标准的 Java Bean 命名规范。如果你的项目使用自定义命名（如 `fetchName()` 而不是 `getName()`），这些方法不会被自动识别。

### Q4: 是否会产生误报？

**A:** 可能会。系统使用基于文本的模式匹配，在某些复杂情况下可能会将非字段声明误识别为字段。建议结合实际代码审查使用。

### Q5: 性能如何？

**A:** 字段检测是轻量级操作，对整体性能影响很小。即使是大型 patch 文件，检测速度也很快。

## 最佳实践

1. **结合代码审查**
   - 自动检测结果应作为参考
   - 重要变更需要人工确认

2. **关注类型变更**
   - 字段类型变更可能影响方法签名
   - 注意泛型参数的变化

3. **检查调用链**
   - 使用影响分析追踪方法调用
   - 识别所有受影响的代码路径

4. **测试覆盖**
   - 为修改的字段添加单元测试
   - 验证 getter/setter 的正确性
