# 通配符导入支持实现说明

## 需求

对于没有明确import的非基础java类，如`Foo foo;`，若代码中有`import bar.*;`，则先在全局索引中查找`bar.Foo`类，若找到了就用作Foo的全限定名，若找不到再使用当前代码的包名。

## 实现方案

### 1. 核心方法

#### `walk_node_for_import_map_with_wildcards`
- 遍历AST节点，提取import语句
- 区分普通导入和通配符导入
- 对于通配符导入（如`import bar.*;`），提取包路径`bar`

#### `build_import_map_with_wildcards`
- 构建两个映射：
  - `import_map`: 简单类名 -> 完整类名（明确导入）
  - `wildcard_imports`: 通配符导入的包列表

#### `resolve_full_class_name_with_wildcard_fallback`
- 解析简单类名为完整类名
- 优先级：
  1. 明确的import语句
  2. 通配符导入的包（使用启发式方法）
  3. 当前包名

#### `resolve_full_class_name_with_wildcards`
- 与上面类似，但使用全局类索引验证
- 只有在全局索引中找到类时才使用通配符导入的包

### 2. 启发式方法 vs 精确方法

#### 启发式方法（当前默认）
- 不需要全局索引
- 假设类在第一个通配符导入的包中
- 优点：简单快速，不需要额外的索引
- 缺点：可能不够精确，特别是有多个通配符导入时

#### 精确方法（可选）
- 需要全局类索引
- 在通配符导入的包中查找类是否存在
- 优点：更精确
- 缺点：需要先构建全局索引

### 3. 修改的方法

- `extract_field_types`: 使用通配符导入解析字段类型
- `extract_method_parameter_types_with_wildcards`: 解析方法参数类型
- `extract_local_variable_types_with_wildcards`: 解析本地变量类型

## 使用示例

### 基本用法（启发式方法）

```rust
let parser = JavaParser::new().unwrap();
let result = parser.parse_file_with_global_types(content, file_path, &global_return_types);
```

这会自动使用启发式方法处理通配符导入。

### 精确方法（使用全局索引）

```rust
// 构建全局类索引
let mut global_class_index = rustc_hash::FxHashMap::default();
global_class_index.insert("bar.Foo".to_string(), "bar.Foo".to_string());

// 使用全局索引解析
let result = parser.parse_file_with_global_types_and_classes(
    content, 
    file_path, 
    &global_return_types,
    &global_class_index
);
```

## 注意事项

1. **启发式方法的限制**：当有多个通配符导入时，所有未明确导入的类都会使用第一个通配符导入的包。这在某些情况下可能不正确。

2. **性能考虑**：启发式方法不需要全局索引，因此更快。精确方法需要先构建全局索引，但更准确。

3. **向后兼容**：现有的API保持不变，新功能通过新的方法提供。

## 未来改进

1. 在两遍索引的第二遍中使用全局类索引，提供更精确的解析
2. 支持静态导入（`import static`）
3. 支持按需导入多个类（`import bar.{Foo, Bar}`）
